Melhoria 1 — Queue semantics para o loop cognitivo (prioridade máxima)
Esse é o gap mais crítico e mais fácil de resolver. Atualmente, o que acontece no Calosum se o usuário enviar uma segunda mensagem enquanto um group turn do GEA está rodando? Provavelmente a mensagem é perdida, processada fora de ordem, ou cria uma condição de corrida no CognitiveWorkspace.
O OpenClaw resolve isso com três modos de fila: steer injeta a nova mensagem no próximo boundary do LLM sem cancelar o run atual; collect segura mensagens até o run terminar; followup enfileira um novo run completo. Para o Calosum, o steer é especialmente poderoso — imagine um group turn GEA onde um novo input do usuário pode ser injetado como contexto de steering antes da próxima chamada ao modelo de scoring, sem derrubar as três variantes que já estão sendo geradas.
A implementação concreta seria adicionar um SessionLane ao CalosumAgent.process_turn(), com um asyncio.Queue por session key. O GEAReflectionController precisaria expor um ponto de injeção de steering entre a geração de variantes e o scoring final.

Melhoria 2 — Session key hierarchy e isolamento (prioridade máxima)
O Calosum provavelmente usa um session_id simples como string opaca. O problema é que isso não diz nada sobre quem está falando, de qual canal, ou se pessoas diferentes devem compartilhar contexto. O OpenClaw prova que essa decisão precisa ser tomada explicitamente.
O padrão a adotar é a key hierárquica: agent:<agentId>:<channel>:direct:<peerId> para DMs isoladas, ou agent:<agentId>:main para single-user. Isso tem um impacto direto no Calosum: o DualMemorySystem vai poder ser consultado com a sessionKey correta, e o HeuristicVerifier vai ter contexto separado por remetente. Sem isso, dois usuários usando o mesmo Telegram bot compartilham memória episódica — o que é um problema de segurança além de cognitivo.
A manutenção de sessões (pruning por idade, limite de entradas, rotação do arquivo de store) também deve ser implementada. O Calosum armazena JSONL em .calosum-runtime/, mas sem nenhuma política de limpeza você vai acumular gigabytes em produção.

Melhoria 3 — DM security model (urgente para produção)
O Telegram adapter do Calosum provavelmente aceita mensagens de qualquer pessoa que saiba o token do bot. O OpenClaw resolve isso com três políticas: pairing (remetentes novos recebem um código e ficam bloqueados até aprovação), allowlist (lista explícita de IDs permitidos), e open (aceita qualquer um, opt-in explícito).
Para o Calosum, o channel_telegram.py precisa de um DmPolicy implementado antes do UserTurn ser enfileirado. Isso também abre a porta para múltiplos usuários aprovados com sessões isoladas.

Melhoria 4 — Compaction pipeline conectado ao NightTrainer (médio prazo, alto impacto)
Esse é o mais elegante do ponto de vista cognitivo. O OpenClaw tem um mecanismo de compactação que, quando o contexto da sessão se aproxima do limite de tokens, dispara um "silent memory flush turn" — o modelo escreve notas duráveis no workspace antes da compactação. Traduzindo para o Calosum: quando o DualMemorySystem detecta que a memória episódica está próxima do limite, um turno silencioso poderia fazer o LeftHemisphere escrever resumos densos no JSONL antes da compactação.
A conexão poderosa é usar esse evento de compactação como gatilho para o NightTrainer. Quando uma sessão é compactada, você tem um fragmento de conversa completo e coerente — exatamente o material ideal para um ciclo de DSPy de auto-melhoria ou para gerar exemplos de fine-tuning LoRA. O OpenClaw tem os hooks before_compaction e after_compaction para isso. No Calosum, o InternalEventBus já existe — basta emitir um evento SESSION_COMPACTED que o NightTrainer escuta.

Melhoria 5 — WebSocket control plane (médio prazo, mudança estrutural)
O Calosum usa FastAPI REST. Para a UI de telemetria, isso significa polling. O OpenClaw prova que o padrão certo é um servidor WebSocket com eventos tipados: lifecycle (start/end/error), assistant (deltas de streaming), tool (start/update/end), cada um com seq para detectar gaps.
Para o Calosum, os eventos cognitivos mapeiam diretamente: right_hemisphere_event (salience calculado, labels emocionais), bridge_event (soft prompts gerados, parâmetros de neuroplasticidade), gea_variant_event (score de cada variante: analitico/empatico/pragmatico), verifier_event (crítica recebida, loop de reparo iniciado). A UI de telemetria poderia mostrar esses eventos em tempo real sem polling, o que é especialmente valioso durante group turns onde três variantes estão sendo avaliadas em paralelo.
A implementação seria uma refatoração do api.py para adicionar um endpoint WebSocket ao lado dos endpoints REST existentes, usando o CognitiveTelemetryBus como fonte de eventos.

Melhoria 6 — Hook points formalizados (quick win)
O Calosum já tem um InternalEventBus e um CognitiveTelemetryBus. O que falta é tornar esses pontos de extensão parte da API pública e documentada. O OpenClaw tem 15 hooks nomeados com semântica clara: before_model_resolve (muda o modelo antes da chamada), before_prompt_build (injeta contexto no system prompt), before_tool_call / after_tool_call (intercept de tools), tool_result_persist (transforma resultados antes do JSONL), session_start / session_end.
Para o Calosum, equivalentes naturais seriam: before_right_hemisphere (pode modificar o input antes da percepção), before_bridge (ajusta parâmetros do tokenizer), before_left_hemisphere (injeta contexto extra no raciocínio), before_lambda_exec (pode modificar parâmetros de tools), after_lambda_exec (pode transformar resultados), before_gea_scoring (pode adicionar critério de avaliação). Isso tornaria o Calosum extensível por plugins externos sem modificar o core — exatamente a promessa do Ports and Adapters que você já segue.
